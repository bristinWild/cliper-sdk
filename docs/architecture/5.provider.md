# Providers

> Cliper understands your repository.
>
> Providers decide where that understanding lives.

---

# Why Providers Exist

Building repository intelligence is only half of the problem.

Once Cliper understands a repository, that knowledge needs a permanent home.

Without storage, every conversation starts from zero.

The provider layer allows Cliper to persist repository memories so they can be recalled later without rebuilding everything again.

---

# Real World Analogy

Imagine spending weeks studying for an exam.

After every study session, you throw away all your notes.

The next day you begin again.

That would be incredibly inefficient.

Instead, you keep your notes somewhere safe.

The next time you study, you continue from where you left off.

Providers work exactly like those notebooks.

They store everything Cliper learns about a repository.

---

# Responsibilities

A provider is responsible for:

- Storing memories
- Updating memories
- Searching memories
- Retrieving context
- Managing relationships

Cliper itself never cares where the memories are stored.

It only knows how to produce them.

---

# Architecture

```
Repository

      │

      ▼

Scanner

      │

      ▼

Memory Engine

      │

      ▼

Knowledge Graph

      │

      ▼

Provider

      │

      ▼

Persistent Memory
```

Providers sit at the final stage of the pipeline.

---

# Provider Interface

Every provider follows the same contract.

```ts
interface MemoryProvider {

    upload()

    search()

    isConfigured()

}
```

Because every provider implements the same interface, switching storage backends does not affect the rest of Cliper.

---

# Current Provider

Today Cliper ships with one provider:

## Cognee

Cognee acts as the persistent memory backend.

During

```
cliper init
```

Cliper automatically:

- scans the repository
- builds structured memories
- uploads those memories
- triggers knowledge extraction
- prepares the repository for semantic retrieval

From that point onward, the repository has a persistent memory.

---

# Why Cognee?

Cognee already provides an excellent foundation for persistent AI memory.

It supports:

- Knowledge Graph construction
- Semantic Search
- Memory persistence
- Relationship extraction
- Long-term recall

Rather than reinventing memory infrastructure, Cliper focuses on producing high-quality repository knowledge.

Cognee focuses on storing and retrieving it efficiently.

The two systems complement each other.

---

# Current Upload Flow

```
cliper init

      │

Repository Scanner

      │

Memory Builder

      │

Structured Memories

      │

Cognee Provider

      │

Cognee Cloud

      │

Knowledge Graph
```

Each memory is uploaded independently.

Once every memory has been uploaded, Cliper requests Cognee to build the repository knowledge graph.

---

# Why Chunked Uploads?

Large repositories can contain thousands of files.

Uploading everything as a single document would quickly exceed language model context limits.

Instead, Cliper uploads memories individually.

For example:

```
Repository Memory

↓

File Memory

↓

Commit Memory

↓

Dependency Memory

↓

Release Memory

↓

Timeline Memory
```

Each memory is self-contained.

This makes uploads:

- smaller
- faster
- easier to update
- easier to debug

# Future Providers

The provider architecture is intentionally storage-agnostic.

Future providers may include:

- Local Memory
- SQLite
- PostgreSQL
- Neo4j
- Memgraph
- Redis
- Vector Databases
- Custom Enterprise Providers

Users will be able to choose the provider that best fits their workflow.

---

# Why This Matters

The scanner understands code.

The memory engine structures it.

The knowledge graph connects it.

The provider preserves it.

Without providers, repository intelligence disappears every time Cliper exits.

Providers transform repository understanding into long-term knowledge.

---

# Looking Ahead

Today, providers primarily store repository memories.

In future versions they will also synchronize:

- Pull Requests
- Issues
- Discussions
- Releases
- Documentation
- Architecture Decision Records
- CI/CD Events

Eventually, a provider will maintain the complete history of a software project—not just its source code.

This enables Cliper to become a true long-term memory system for software engineering.
