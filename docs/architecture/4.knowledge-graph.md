# Knowledge Graph

> A repository is more than a collection of files.
>
> It is a network of connected ideas.

---

# Why a Knowledge Graph?

Traditional code search treats a repository like a folder of documents.

You search for text.

Open files.

Read code.

Search again.

Developers don't think this way.

When working on a project, we think in concepts:

- Authentication
- Payments
- Wallets
- APIs
- Database
- Deployment

These concepts are connected.

Cliper captures those connections by transforming repository information into a Knowledge Graph.

---

# Real World Analogy

Imagine visiting a city you've never seen before.

Without a map, every building looks isolated.

You may know where the airport is.

You may know where a restaurant is.

But you don't know how everything connects.

Now imagine opening Google Maps.

Suddenly you understand:

- Roads
- Highways
- Metro lines
- Neighbourhoods
- Landmarks

The city hasn't changed.

Only your understanding has.

A Knowledge Graph does the same for software repositories.

Files become buildings.

Relationships become roads.

The graph becomes the map.

# Why Relationships Matter

Suppose someone asks:

> "Why is this package used?"

A traditional code search returns every file mentioning it.

You still need to manually investigate.

With a Knowledge Graph, Cliper can traverse:

```
Package

↓

Imported by

↓

wallet.ts

↓

Modified in Commit

↓

Released in v0.2.0

↓

Part of Authentication Module
```

Instead of finding information, it understands the context surrounding that information.

# Why Not Just Vector Search?

Vector databases are excellent at finding similar text.

Knowledge Graphs answer a different question.

Instead of asking:

> "Which files mention authentication?"

You can ask:

> "Which release introduced authentication?"

Or:

> "Which package is responsible for authentication?"

Those answers require relationships, not similarity.

Vector Search tells you **where**.

Knowledge Graphs tell you **why**.

Cliper combines structured memories with relationships to provide both.

# Incremental Growth

The Knowledge Graph evolves alongside the repository.

Every new commit can introduce:

- new files
- new dependencies
- new responsibilities
- new releases
- new relationships

Rather than rebuilding understanding from scratch, Cliper continuously expands the graph.

---

# Future Graph Expansion

The current graph focuses on repository structure.

Future versions will extend the graph with development workflow knowledge.

Upcoming node types include:

- Issues
- Pull Requests
- Discussions
- Contributors
- Documentation
- ADRs (Architecture Decision Records)

This allows Cliper to answer questions such as:

- Why was this change made?
- Which discussion introduced this architecture?
- Which issue is responsible for this implementation?
- Which developer owns this module?

---

# The Bigger Vision

The long-term goal isn't simply to index repositories.

It's to build a persistent memory system for software projects.

A repository should not have to explain itself every time an AI assistant asks a question.

Instead, it should have an evolving memory that grows with every commit, release, discussion, and architectural decision.

That memory is represented as a Knowledge Graph.

This transforms software repositories from static collections of files into living systems of interconnected knowledge.
