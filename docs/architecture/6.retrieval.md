# Retrieval

> Building memories is only useful if they can be recalled.

---

# The Retrieval Problem

Most AI assistants start every conversation from scratch.

Every question requires:

- scanning files
- reading source code
- rebuilding context

This becomes slower as repositories grow.

Cliper approaches retrieval differently.

Instead of rebuilding understanding, it remembers it.

---

# Real World Analogy

Imagine asking a senior engineer:

> "How does authentication work?"

They don't reopen every source file.

They answer from memory.

Only if necessary do they inspect the code.

Cliper follows the same philosophy.

---

# Retrieval Pipeline

```
User Question

      │

      ▼

Provider

      │

      ▼

Relevant Memories

      │

      ▼

Knowledge Graph Traversal

      │

      ▼

Relevant Context

      │

      ▼

LLM
```

The model receives understanding instead of raw source code.

---

# What Gets Retrieved?

Depending on the question, Cliper may retrieve:

- Repository Memory
- File Memories
- Dependency Memories
- Commit Memories
- Release Memories
- Timeline Memories
- Architecture Memories
- Responsibility Memories

Only the memories relevant to the question are returned.

---

Question

---

# Why This Is Faster

Traditional workflow

```
Question

↓

Repository Scan

↓

File Reading

↓

Reasoning
```

Cliper workflow

```
Question

↓

Memory Search

↓

Graph Traversal

↓

Reasoning
```

The expensive repository analysis has already happened.

---

# Persistent Context

Because memories persist, Cliper can answer questions across multiple sessions.

It doesn't need to rediscover repository knowledge every time.

The repository accumulates experience instead of forgetting it.

# Long-Term Vision

Retrieval is the final step that makes repository memory useful.

The scanner learns.

The memory engine remembers.

The provider stores.

The retrieval engine recalls.
