# Repository Scanner

> "Before an AI can remember a project, it must first understand it."

---

# The First Day at a New Job

Imagine joining a company as a software engineer.

Nobody expects you to immediately start writing code.

Instead, your first few hours look something like this.

You...

- clone the repository
- explore the folders
- open the README
- inspect the package manager
- understand the architecture
- read recent commits
- identify important modules

Only after that do you begin writing code.

Cliper follows exactly the same process.

The Repository Scanner is responsible for learning everything about a project before any memories are created.

---

# The Scanner Pipeline

```
Repository

        │

        ▼

File Tree Scanner

        │

        ▼

File Content Scanner

        │

        ▼

Git Scanner

        │

        ▼

Dependency Scanner

        │

        ▼

Semantic Scanner

        │

        ▼

Gap Detection

        │

        ▼

Memory Builder
```

Each scanner has a single responsibility.

Instead of one enormous parser trying to understand everything, Cliper divides repository understanding into independent stages.

---

# File Tree Scanner

```
scanner/fileTree.ts
```

The first thing Cliper learns is the repository structure.

Example

```
src/

commands/

scanner/

providers/

sdk/

README.md

package.json
```

Just like a developer quickly understands a city by looking at a map, Cliper understands a project by first learning its layout.

This produces the repository skeleton.

---

# File Content Scanner

```
scanner/fileContent.ts
```

Once the structure is known,

Cliper starts reading source files.

For every file it extracts

- relative path
- source code
- language
- size
- metadata

This becomes the raw material for later analysis.

Think of it as reading every chapter of a book before writing the summary.

---

# Git Scanner

```
scanner/gitContext.ts
```

A repository isn't just code.

It's history.

The Git Scanner extracts

- current branch
- latest commit
- recent commits
- repository status
- uncommitted changes

This allows Cliper to understand how the project evolved over time.

Two repositories may contain identical code today but completely different histories.

History matters.

---

# Dependency Scanner

```
scanner/dependencies.ts
```

Individual files rarely work alone.

This scanner discovers relationships such as

```
AuthController

↓

AuthService

↓

UserRepository
```

Instead of isolated files,

Cliper begins seeing a connected system.

Dependencies eventually become part of the repository knowledge graph.

---

# Semantic Scanner

```
scanner/semanticLabels.ts
```

Source code contains meaning.

Not every file is "just another TypeScript file."

Some are

- authentication
- payments
- blockchain
- database
- API
- configuration

The Semantic Scanner identifies these higher-level concepts.

Instead of remembering filenames,

Cliper remembers intent.

Humans think in concepts.

AI should too.

---

# Gap Detection

```
gaps/detector.ts
```

Repositories are rarely perfect.

The Gap Detector searches for things such as

- TODOs
- undocumented modules
- missing documentation
- unfinished implementations
- architectural inconsistencies

Rather than hiding weaknesses,

Cliper records them as first-class knowledge.

Knowing what is missing is often as valuable as knowing what exists.

---

# Why Multiple Scanners?

Imagine asking one person to

- read every file
- understand Git
- infer architecture
- classify modules
- detect missing documentation

That person would become slow, complex, and difficult to maintain.

Instead,

Cliper uses specialized scanners.

Each scanner answers one question.

```
Where are the files?

↓

What do they contain?

↓

How are they connected?

↓

What do they mean?

↓

What is missing?
```

Simple components.

Clear responsibilities.

Easy to extend.

---

# Scanner Independence

Every scanner is isolated.

```
scanner/

fileTree.ts

fileContent.ts

gitContext.ts

dependencies.ts

semanticLabels.ts
```

Adding a new scanner never requires rewriting the existing ones.

Future scanners may include

- GitHub Issues
- Pull Requests
- Discussions
- Jira
- Linear
- Slack
- Documentation

The pipeline simply grows.

---

# The Philosophy

Most repository analysis tools answer

> "What files exist?"

Cliper asks

> "What does this repository know?"

That difference is what transforms source code into memory.

The scanner doesn't create memories.

It creates understanding.

Everything else builds upon that foundation.
