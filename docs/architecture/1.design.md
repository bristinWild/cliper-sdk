# Cliper Design Philosophy

> **Every repository already contains knowledge. Cliper's job is to discover it, structure it, and make it searchable.**

---

# The Problem

Modern codebases grow much faster than documentation.

A repository may contain:

- hundreds of source files
- thousands of functions
- months of Git history
- architectural decisions
- hidden dependencies
- undocumented assumptions

Yet most of this knowledge lives only inside the code.

When a new developer joins a project, they usually ask questions like:

- Where does authentication happen?
- Which file owns payments?
- What changed in the last release?
- Which module depends on this package?
- Why was this function written?

Unfortunately, answering these questions often requires manually reading dozens of files or asking another engineer.

The repository already knows the answers.

It simply has no memory.

---

# The Idea

Cliper treats a repository like a living system rather than a collection of files.

Instead of generating one massive context document, Cliper decomposes the repository into small pieces of structured knowledge called **memories**.

Each memory represents a single fact.

Examples include:

- a source file
- a dependency
- a Git commit
- a release
- a package
- a repository statistic
- an architectural component
- a responsibility
- a detected knowledge gap

Together these memories form a searchable knowledge graph.

---

# Real World Analogy

Imagine entering a huge library.

Traditional documentation gives you a single book describing the library.

Cliper builds the entire catalogue.

Every shelf,
every book,
every author,
every relationship

is indexed.

Instead of asking:

> "Read the whole library."

you simply ask:

> "Show me every book related to networking."

The information already existed.

Cliper simply organized it.

---

# Why Structured Memory?

Large Language Models perform better when information is:

- small
- specific
- interconnected

instead of being buried inside one enormous prompt.

Rather than uploading one 100k-token document, Cliper creates hundreds of independent memories.

For example:

```
Repository
    ↓
Authentication Module
    ↓
Login Service
    ↓
JWT Dependency
    ↓
Recent Commit
```

This makes retrieval significantly more precise.

---

# Repository as a Knowledge Graph

Instead of storing plain text, Cliper stores relationships.

Example:

```
Repository
    │
    ├── contains
    │      │
    │      ├── File
    │      ├── Package
    │      ├── Commit
    │      └── Release
    │
    ├── depends on
    │
    ├── modified by
    │
    ├── documented by
    │
    └── responsible for
```

The repository becomes a connected graph rather than isolated files.

---

# Why Cognee?

Cliper focuses on understanding repositories.

Cognee focuses on storing and retrieving structured memory.

Instead of reinventing memory storage, Cliper generates high-quality repository intelligence and delegates persistence to Cognee.

This separation keeps both systems focused on what they do best.

---

# Repository Initialization

Running

```bash
cliper init
```

starts a complete repository analysis.

The scanner extracts:

- repository structure
- source files
- dependencies
- git history
- releases
- project statistics
- architecture
- responsibilities
- semantic labels
- detected gaps

These are transformed into structured memories.

If a Cognee provider is configured, the memories are automatically synchronized into the user's private Cognee dataset.

The repository now has persistent memory.

---

# Retrieval

Once memories exist, Cliper no longer needs to search the repository from scratch.

Instead it retrieves relevant memories.

For example:

> "Show me payment related files."

returns payment memories.

> "Which release introduced JWT?"

returns release memories.

> "Why does this module exist?"

returns architecture memories.

The system searches knowledge instead of raw text.

# Design Principles

Cliper is built around a small set of principles.

## Repository First

The repository is the source of truth.

Knowledge should be extracted rather than manually maintained.

---

## Structured Over Raw

Small connected memories are more valuable than one enormous document.

---

## Relationships Matter

Understanding a repository requires understanding how components connect.

The graph is as important as the nodes.

---

## Provider Agnostic

Memory generation is independent from memory storage.

Cognee is the first provider, but not the only possible destination.

---

## Incremental Knowledge

A repository evolves every day.

Its memory should evolve with it through commits, releases, pull requests, and discussions.

---

# In One Sentence

**Cliper transforms software repositories into living knowledge graphs that can be understood, searched, and eventually conversed with from anywhere.**
